FAQ — problèmes courants

Les situations les plus fréquentes et comment les résoudre.

Aucun renderer / interface ne se charge

Les plugins .so sont cherchés dans le dossier ./plugins relatif au répertoire courant. Lancez toujours le programme depuis la racine du projet :

cd /chemin/vers/raytracer_V2
./raytracer tests/configs/subject.json

Si ./plugins est vide, (re)compilez avec make. Un rendu correct affiche : Loaded 2 renderer plugin(s).

« Failed to load font »

L’interface charge assets/font.ttf par un chemin relatif. Lancez également depuis la racine du projet, où se trouve le dossier assets/.

« Setting vertical sync not supported »

Message informatif de SFML, fréquent sous WSL où le rendu OpenGL est logiciel (Mesa/llvmpipe). Il est sans conséquence sur le rendu.

Un rapport « LeakSanitizer: detected memory leaks » à la fermeture

Le projet est compilé avec AddressSanitizer (-fsanitize=address), qui inclut un détecteur de fuites exécuté à la fermeture. Si le rapport pointe des adresses <unknown module> issues de calloc sans aucune image de la pile dans le code du projet, il s’agit d’une petite allocation unique du pilote graphique Mesa/swrast (rendu logiciel sous WSL) — extérieure au moteur.

Ces allocations de pilote ne sont pas symbolisées et ne peuvent donc pas être filtrées par nom. Pour ne détecter que les fuites du code applicatif (qui, elles, doivent être nulles), il suffit d’auditer en mode headless qui n’ouvre pas de contexte graphique :

# Audit des fuites du moteur (sans interface)
ASAN_OPTIONS=detect_leaks=1 ./raytracer -r sortie.png tests/configs/subject.json

# Désactiver le rapport de fuites pour une session graphique
ASAN_OPTIONS=detect_leaks=0 ./raytracer tests/configs/subject.json

Les autres vérifications d’AddressSanitizer (use-after-free, débordements, double-free…) restent actives dans tous les cas.

« object "…" : … » ou « Missing camera / objects »

Le SceneParser signale les problèmes du fichier JSON sur la sortie d’erreur. Vérifiez que :

  • le fichier est un JSON valide et se termine par .json ;
  • les sections camera et objects sont présentes ;
  • la caméra contient resolution ([largeur, hauteur]) et fieldOfView ;
  • chaque objet possède un type valide (voir les types).

Les clés inconnues d’un objet sont silencieusement ignorées. Un paramètre mal orthographié (ex. normal au lieu de axis pour un plan) n’est donc pas signalé : l’objet prend simplement ses valeurs par défaut.

Mon matériau ne s’applique pas

Un objet référence un matériau par son nom ("material": "Mon matériau"). Ce nom doit correspondre exactement à une entrée de la section materials. Sinon, le parseur signale Unknown material "…".

Les objets métalliques ou en verre paraissent très sombres

Le moteur ne dispose pas d’environnement (skybox) : un matériau très réfléchissant ou transparent renvoie donc principalement du « vide » sombre là où il n’y a pas de géométrie ou de lumière à réfléchir. Ajoutez de la géométrie et des lumières autour de l’objet, ou réduisez reflectivity/transparency pour un rendu plus lisible.

« --render cannot be combined with --server » et autres erreurs d’options

Certaines options sont incompatibles : --render (headless) ne se combine pas avec --server/--connect, et --server/--connect sont exclusifs. Une option inconnue provoque Error: unknown option '…'. Utilisez --help pour la liste.

Fenêtres / interface sous WSL

Sous WSL, l’interface rend le fenêtrage thread-safe (XInitThreads) et force le rendu OpenGL logiciel pour supporter plusieurs fenêtres (pop-ups) sans planter. Assurez-vous que WSLg (ou un serveur X) est disponible et que DISPLAY est défini.

Un maillage .obj ne se charge pas

Vérifiez que le chemin de la clé file est correct par rapport au répertoire d’où vous lancez le programme (des exemples sont fournis dans tests/obj/). Les faces sont triangulées et un BVH local est construit ; pour de très gros maillages, la première construction peut prendre un instant.
