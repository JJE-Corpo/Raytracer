Accueil / Architecture

Architecture

Le code s’organise autour d’un Core central qui orchestre le chargement de scène, des plugins, le rendu, l’interface optionnelle et le mode cluster optionnel.

Le Core

Le Core (srcs/core/Core.*) coordonne tout le cycle de vie :

  1. charge les plugins (loadPlugins) et les renderers (loadRenderers) ;
  2. analyse le fichier .json et construit la scène via les builders ;
  3. sélectionne le renderer actif ;
  4. construit le BVH et lance le rendu ;
  5. exporte l’image ;
  6. optionnellement, lance la boucle d’événements de l’interface et le rechargement à chaud du fichier de scène (FileObserver).

Le Core gère aussi l’historique (annuler / rétablir) : chaque mutation sérialise un instantané JSON de la scène ; un undo/redo reconstruit la scène à partir de l’instantané voulu.

Système de plugins

Les modules de rendu et d’interface sont des bibliothèques partagées .so chargées à l’exécution depuis ./plugins.

Au démarrage : PluginLoader::load() appelle dlopen et résout deux symboles obligatoires (contrat C ABI) :

extern "C" IPlugin *create_plugin();
extern "C" void      destroy_plugin(IPlugin *plugin);

Sans ces deux fonctions, le plugin est rejeté. Les instances sont stockées dans le loader et interrogées par PluginType (RENDERER, USER_INTERFACE).

PluginTypeRôle
renderer_default.soRENDERERRenderer « Default » — rendu complet de l’image.
renderer_viewport.soRENDERERRenderer « Viewport » — rendu progressif pour l’interface.
user_interface.soUSER_INTERFACEInterface graphique SFML (viewport, panneaux, gizmos).

Les primitives et lumières vivent dans srcs/plugins/… mais sont actuellement compilées dans l’exécutable principal et instanciées via les builders/factories — elles ne sont pas (encore) chargées comme .so dynamiques. L’architecture reste néanmoins prête pour cette évolution.

Construction de la scène

La construction est séparée de l’analyse :

scene.json
   │
   ▼
SceneParser ──► parseCamera
   │        ├─► parseMaterials
   │        └─► parseObjects ──► SceneObjectBuilder ──► PrimitiveBuilder
   │                                                └─► LightBuilder
   ▼
SceneBuilder ──► Scene   (caméra + matériaux + objets, dispatch par ObjectType)
  • SceneObjectBuilder — répartiteur : withType(string) (primitive) ou withType(LightType) (lumière), et transmet les setters communs au builder interne actif.
  • PrimitiveBuilder — construit les primitives concrètes à partir des champs analysés (position, radius, size…).
  • LightBuilder — construit PointLight et DirectionalLight.
  • SceneBuilder — agrège caméra, matériaux et objets, puis répartit selon ObjectType (PRIMITIVE → addPrimitive, LIGHT → addLight).
  • PrimitiveFactory / LightFactory — constructeurs par défaut (par ex. pour la création d’objets depuis l’éditeur).

SceneRegister effectue le chemin inverse : il re-sérialise l’état de la scène en JSON (caméra, environnement, objets, matériaux), permettant l’édition en aller-retour dans l’interface.

Pipeline de rendu

Une image est produite en appliquant la même séquence à chaque pixel :

  1. Rayon caméra — génération d’un rayon primaire P(t) = O + t·D pour le pixel.
  2. Intersection — test contre la géométrie (accélérée par le BVH construit avant le rendu via scene.buildBvh()).
  3. Hit le plus proche — conservation du plus petit t valide.
  4. Ombrage — au point d’impact : calcul de la normale, lecture du matériau.
  5. Lumières & ombres — pour chaque lumière, un rayon d’ombre décalé (H + ε·N) teste la visibilité ; si visible, on évalue diffus et spéculaire.
  6. Accumulation — ambiant + contributions directes, écrêtage dans la plage de sortie.
  7. Sortie — le tampon image est exporté en PNG.

Robustesse numérique : décalage ε pour les rayons secondaires (évite l’acné d’ombre), rejet des dénominateurs proches de zéro (plans/triangles), test cohérent t ∈ [t_min, t_max], vecteurs normalisés pour l’ombrage.

Module cluster

Le ClusterModule (srcs/core/cluster/*) fournit une infrastructure de rendu distribué : un serveur découpe l’image en tuiles et les distribue à des clients connectés qui renvoient leurs pixels. La communication passe par des paquets typés (PacketFactory), par exemple : demande de rejoindre (ClientJoinRequest), envoi de la scène (ServerSceneData), demande de rendu (ServerRenderRequest), remontée d’une tuile (ClientTileData), état du rendu (ServerRenderState).

  • --server [PORT] lance l’interface et démarre le serveur (startServer).
  • --connect IP PORT lance l’interface et rejoint un serveur (joinCluster).

Le rendu est coordonné par ClusterRenderCoordinator et ClusterRenderer.
