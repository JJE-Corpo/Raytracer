Fonctionnalités

Géométrie, éclairage, matériaux et outils interactifs disponibles dans le moteur.

Primitives

Dix types de primitives sont reconnus par le parseur et les builders/factories. Chaque primitive implémente l’interface IPrimitive (test d’intersection, boîte englobante, matériau, propriétés).

Géométrie d’accélération

Deux structures utilitaires soutiennent le rendu : AABB (boîte englobante alignée sur les axes) et BVH (hiérarchie de volumes englobants). Voir la section BVH.

Lumières

Deux types de lumières sont pris en charge par le parseur et les builders. Chaque lumière implémente ILight (intensité, couleur).

LumièretypeParamètresDescription
Ponctuellepoint_lightposition, color, intensityÉmet dans toutes les directions depuis un point.
Directionnelledirectional_lightdirection, color, intensityRayons parallèles (type « soleil »), sans position.

Ombres

Pour chaque lumière et chaque point d’impact, le moteur lance un rayon d’ombre depuis un point légèrement décalé (H + ε·N) vers la lumière. Si un objet occulte le trajet, la contribution directe de cette lumière est ignorée pour ce point ; sinon, les termes diffus et spéculaire sont évalués. Le décalage ε évite l’acné d’ombre (auto-intersection).

Réflexions et transparence

Les champs reflectivity, transparency et ior du matériau contrôlent respectivement la réflexion miroir, la transparence et la réfraction. Ils reposent sur des rayons secondaires partant du point d’impact : rayons de réflexion et de réfraction, lancés avec un décalage ε pour éviter l’auto-intersection.

Accélération BVH

Avant le rendu, la scène construit une hiérarchie de volumes englobants (BVHNode, via scene.buildBvh()). Chaque nœud possède une boîte englobante ; un rayon qui manque la boîte d’un nœud ignore tout son sous-arbre, ce qui réduit fortement le nombre de tests d’intersection. Les maillages disposent en plus d’un BVH local sur leurs triangles.

Maillages et import .obj

Un maillage charge un fichier Wavefront .obj (via ObjParser), en dérive des triangles (MeshTriangle), calcule des normales lisses (moyenne des faces incidentes) et construit un BVH local. Un maillage peut aussi être défini en ligne dans la scène (vertices/faces/normals), ce qui permet de le sérialiser sans fichier externe.

Édition de sommets

Les primitives qui implémentent IEditablePrimitive peuvent voir leurs sommets déplacés directement dans le viewport. Sont éditables :

PrimitiveSommets éditables
Triangleses 3 coins.
Maillageles sommets uniques (soudés) du .obj — déplacer un sommet partagé met à jour toutes ses faces.

Les primitives analytiques (cube, sphère…) ne sont pas éditables au sommet : sélectionnez-les et utilisez Convert to Mesh pour les transformer en maillage triangulé, puis éditez-les. Un cube est converti exactement en 8 sommets / 12 triangles ; les autres formes sont échantillonnées.

Contrôles

  • Entrer / sortir : sélectionner une primitive éditable puis Tab (ou double-clic) ; Tab ou Échap pour sortir.
  • Poignées : chaque sommet est un cercle — bleu au repos, blanc sous le curseur, jaune sélectionné.
  • Déplacement : glisser dans le plan parallèle à l’écran ; maintenir X, Y ou Z pour verrouiller un axe.
  • Saisie précise : le champ Vertex du panneau Object affiche/édite les coordonnées du sommet sélectionné.
  • Navigateur : les flèches < Point N / total > parcourent les sommets un à un.

Les modifications d’un maillage sont enregistrées comme surcharges de sommets (vertex_overrides, en espace-objet) réappliquées après le chargement du .obj ; le fichier .obj n’est jamais modifié. Après chaque édition, le BVH est marqué à reconstruire et le viewport se met à jour.

Rendu en cluster

Le moteur peut répartir un rendu sur plusieurs machines : un serveur découpe l’image en tuiles et les distribue aux clients connectés, qui renvoient leurs pixels. On lance un serveur avec --server [PORT] et on rejoint avec --connect IP PORT. La communication repose sur des paquets typés (rejoindre, envoi de scène, demande de rendu, remontée de tuile, état du rendu). Voir Architecture → Cluster.
